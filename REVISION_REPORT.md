# バックアップ・リストア仕組み 修正レポート

## 1. 修正内容のまとめ
バックアップおよびリストアの仕組みにおいて、セキュリティの向上、コマンドの互換性確保、エラーログの管理、および処理の堅牢性を目的とした修正を行いました。

### セキュリティの強化
- **MySQLパスワードの安全な取り扱い**: 
  - `mysql` や `mysqldump` コマンドの引数に直接パスワードを指定する方式 (`-p'password'`) は、プロセスリスト (`ps`コマンド等) からパスワードが露出するリスクがあるため廃止しました。
  - 代わりに `MYSQL_PWD` 環境変数を使用するように変更しました。これにより、特殊文字を含むパスワードでも安全に処理できるようになりました。
- **設定ファイルの保護**: 
  - `setup.sh` 実行時に、機密情報が含まれる `config.env` のパーミッションが `600` (所有者のみ読み書き可能) であるかを確認し、必要に応じて変更を促すチェック機能を追加しました。

### 互換性と堅牢性の向上
- **OS間コマンド互換性**: 
  - 一部の Linux ディストリビューションで挙動が異なる場合がある `zcat` コマンドの使用を避け、より汎用的な `gunzip -c` に統一しました。
- **リストア処理の修正**: 
  - `restore.sh` 内で `eval` を使用してコマンドを実行する際、パスにスペースが含まれる場合のクォート漏れなどを修正し、予期せぬ動作を防ぐようにしました。
- **テストリストアの安全性**:
  - `test_restore.sh` において、実行後にパスワードを含む環境変数を確実に `unset` するようにし、環境変数が残存するリスクを排除しました。

### 運用性の改善
- **ログ管理の適正化**: 
  - エラーログ出力において、`2>>` (追記) から `2>` (最新エラーによる上書き) に変更しました。これにより、バックアップ失敗時に過去の無関係なエラーに埋もれることなく、最新の失敗原因を即座に特定できるようになりました。
  - **logrotate の導入**: `setup.sh` にて `/etc/logrotate.d/backup-${SERVICE_NAME}-${ENVIRONMENT}` を自動作成するようにし、ログファイルの肥大化によるディスク圧迫を防ぐ仕組みを追加しました。
- **マルチサービス対応**: `SERVICE_NAME` 変数を導入し、同一サーバー内の複数サービスを個別に管理（バックアップファイルの分離、Cron/Logrotate設定の分離）できるようにしました。

## 2. 運用リスクと対応案
将来的な不具合や運用上のトラブルを防ぐためのリスク特定と、その対応策を提案します。

### リスク1: ログファイルの肥大化
- **内容**: `/var/log/backup/*.log` が時間経過とともに増え続け、ディスク容量を圧迫する可能性があります。
- **対応**: `setup.sh` にて、OS標準の `logrotate` 設定 (`/etc/logrotate.d/backup-${SERVICE_NAME}-${ENVIRONMENT}`) を自動生成するように対応しました。
  - 設定内容: 毎日実行、14世代保持、圧縮有効。

### リスク2: バックアップ先ストレージの容量不足
- **内容**: ローカルの `/backup` ディレクトリやクラウドストレージの容量が不足すると、バックアップが正常に完了しなくなります。
- **対応案**: 
  - 定期的な容量監視（ZabbixやMackerel等の監視ツール、または単純な `df` チェックスクリプトのCron実行）を導入し、閾値を超えた場合に通知する仕組みを検討してください。

### リスク3: ネットワーク遮断による同期失敗
- **内容**: `rclone` によるクラウド同期中にネットワークが切断された場合、その時点のファイルが同期されません。
- **対応案**: 
  - `backup_sync.sh` には既にリトライ設定が含まれていますが、重要な環境では定期的に `rclone check` を実行し、ローカルとリモートの整合性を検証するジョブを追加することを検討してください。

- セキュリティ上の修正（環境変数への移行やパーミッションチェック）が意図通り動作することを確認しました。
